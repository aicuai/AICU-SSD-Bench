# ICY DOCK リムーバブルベイ性能分析 (Drive D vs G)

> Samsung 9100 PRO 8TB を直接接続 (D:) と ICY DOCK リムーバブルベイ経由 (G:) で比較

## 結論

**ICY DOCK リムーバブルベイは Gen5 NVMe の性能を損なわない。**
Raw disk speed で 1.3% 差、15.6GB モデルのロードでも 3% 差に留まる。
GPU バウンドのワークロード（推論・生成）では完全に同等。

---

## Raw Disk Speed (Ex0)

| メトリクス | D: (直接) | G: (ICY DOCK) | 差 |
|-----------|-----------|---------------|-----|
| Sequential Read (median) | 7,037.8 MB/s | 6,945.8 MB/s | 1.3% |
| Sequential Write (median) | 2,535.1 MB/s | 2,529.8 MB/s | 0.2% |

同一の Samsung 9100 PRO。差はシステムノイズの範囲。

## ワークロード別比較

### GPU バウンド（ストレージ差なし）

| ワークロード | D: | G: | 差 | 備考 |
|---|---|---|---|---|
| ComfyUI 画像生成 (cold) | 7.18s | 7.18s | 同等 | モデル一括ロード→GPU推論 |
| ComfyUI LTX 2.3 t2v (cold) | 98.4s | 98.7s | 0.4% | 同上 |
| Ollama tokens/sec | 185.87 | 185.33 | 0.3% | 推論速度は完全同等 |

### I/O バウンド（モデルサイズに比例した微小差）

| ワークロード | D: | G: | 差 | モデルサイズ |
|---|---|---|---|---|
| Moshi LM ロード | 7.50s | 7.71s | G が 3% 遅い | 15.6 GB |
| Ollama ロード (qwen3:8b) | 3.09s | 2.66s | G が 14% 速い※ | ~8 GB |
| Qwen3-TTS モデルロード | 6.42s | 6.55s | 同等 | ~1.7 GB |

※ Ollama ロードの D vs G 差はキャッシュ状態の影響が大きい。

### 見かけ上の差があるが実はストレージ無関係

| ワークロード | D: | G: | 見かけの差 | 真因 |
|---|---|---|---|---|
| Ollama コード生成 | 9.37s | 13.78s | D が 47% 速い | LLM 出力トークン数のばらつき (D:1494 vs G:2238 tokens) |
| Qwen3-TTS TTFA | 9.80s | 7.51s | G が 23% 速い | 合成処理の非決定性 (5.5~13.2s の範囲) |

## Qwen3-TTS: RAM ディスク高速化の可能性

### 現状の問題

Qwen3-TTS のボイスサンプリング機構は、音声合成の各ステップで**参照 MP3 ファイルを再帰的にディスクから読み込む**。
このため、合成時間のばらつきが大きく（5.5~13.2s）、長文合成では累積的にディスク I/O が発生する。

### 仮説

参照 MP3 をオンメモリ（RAM ディスクまたは tmpfs）に配置することで：

1. **ディスク I/O レイテンシの排除** — NVMe でも ~100μs の read latency が、メモリなら ~100ns に
2. **OS ファイルキャッシュのばらつき排除** — RAM ディスクなら常にメモリ上に保証
3. **合成時間のばらつき低減** — 5.5~13.2s の range が狭まる可能性

### 検証実験案

```
Ex11: Qwen3-TTS RAM ディスク比較
  - 条件A: 参照 MP3 を NVMe (D:) に配置（現状）
  - 条件B: 参照 MP3 を RAM ディスク (ImDisk/imdisk) に配置
  - 条件C: 参照 MP3 をプリロードしてメモリ上にキャッシュ（Python io.BytesIO）
  - 各条件 × 5回 × 5テキスト
  - 計測: TTFA, 各テキスト合成時間, 合計時間, ばらつき (std dev)
```

### 期待される結果

- **短文 (S1)**: 効果大 — I/O 比率が高い
- **長文 (L1, L2)**: 効果小 — GPU 推論時間が支配的
- **ばらつき**: RAM ディスクで有意に低減されるはず

### ComfyUI / Ollama では効果なし

- ComfyUI: モデルを一括ロードした後は全て GPU VRAM 上で完結
- Ollama: mmap でモデルをマッピング、推論中のディスク再読み込みはない
- → **推論中にディスク I/O が発生するワークロードにのみ RAM ディスクは有効**

## まとめ

| 観点 | 結論 |
|------|------|
| ICY DOCK オーバーヘッド | 実質ゼロ (1.3%) |
| GPU バウンドワークロード | ストレージ完全無関係 |
| 大規模モデルロード (15GB+) | disk speed 差がそのまま反映 (~3%) |
| RAM ディスク高速化 | Qwen3-TTS など推論中 I/O があるケースで有効な可能性 |
